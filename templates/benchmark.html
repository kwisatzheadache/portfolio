2012-2015 Benchmark Construction and Others

Carpentry and software development may seem like vastly different fields, but there are actually many similarities between the two. One of the most notable parallels is the fact that, when you start a remodeling project or a new software project, you often don't know exactly what you're dealing with until you start digging in. In the case of carpentry, you might start a remodeling project with a plan to remove a wall, only to discover that the wall is load-bearing and you'll need to come up with a different solution. Similarly, when you start working on a new software project, you might begin with a set of assumptions about the problem you're solving, only to realize that those assumptions were incorrect and you need to pivot your approach.

As an example, I once remodeled a bathroom that had a wall with 0 continuous studs. This meant that I had to come up with a completely new plan for reinforcing the wall to make sure it could support the weight of the shower and other fixtures.

Both carpentry and software development require the ability to hold complex structures in your mind and consider all the different ways to fit them together. In carpentry, you might need to think about how to frame a room or build a staircase, while in software development, you might need to consider how to structure your code to make it maintainable and scalable.

Additionally, building codes in carpentry are not that different from software requirements. Just as building codes exist to ensure that structures are safe and up to code, software requirements exist to ensure that the software you're building meets certain standards and fulfills its intended purpose. In both cases, there are often strict guidelines that need to be followed to ensure that the end result is functional and reliable.
